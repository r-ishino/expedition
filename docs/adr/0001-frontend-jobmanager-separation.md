# ADR-0001: フロントエンドとジョブマネージャーの分離

## ステータス

採用

## コンテキスト

Expedition は Jira タスクを起点に複数の AI コーディングエージェント（Claude Code, Devin）を並列実行するオーケストレーションツールである。

システムには大きく2つの責務がある:

1. **UI の提供** — カンバン形式のタスク一覧、操作画面、PR/CI 状態の可視化
2. **ジョブの実行管理** — git worktree の作成、AI エージェントプロセスの起動・監視、CI 結果の待機とリトライ

これらを単一の Next.js アプリケーションに統合するか、別サービスとして分離するかを検討した。

## 決定

**frontend（Next.js）と job-manager（Hono）を別サービスとして分離する。**

- `apps/frontend` — UI の提供。カンバン表示、操作画面
- `apps/job-manager` — ジョブのライフサイクル管理。エージェント起動、worktree 管理、CI 監視

## 理由

### 1. プロセスの実行時間

AI エージェントのセッションは数分〜数十分に及ぶ。Next.js の Server Actions / Route Handlers はリクエスト・レスポンスモデルが前提であり、長時間実行プロセスの保持には向かない。job-manager は常駐プロセスとしてこれらを管理する。

### 2. ライフサイクル管理の複雑さ

Phase 2 で求められる要件:

- 複数エージェントの並列実行（worktree × プロセスの組み合わせ）
- プロセスの状態追跡（開始・完了・エラー・タイムアウト）
- CI fail 時の自動リトライ（ポーリング/webhook → 再実行のループ）

これらはリクエスト単位ではなく、サービス全体で状態を保持する常駐型の処理である。Next.js のリクエストスコープでは横断的な状態管理が困難になる。

### 3. デプロイの独立性

- フロントエンドの再デプロイ時にバックグラウンドジョブが中断されない
- job-manager のみスケールアウトできる（エージェント数の増加に対応）
- 障害の分離 — UI の障害がジョブ実行に影響しない

### 4. 開発の関心の分離

フロントエンドは React/UI に集中し、job-manager はプロセス管理・外部 API 連携に集中できる。それぞれの変更が互いに影響しにくい。

## 却下した選択肢

### Next.js への統合

- **メリット**: サービスが1つで済み、初期の開発・デプロイが簡単
- **却下理由**: Phase 2（並列実行・CI リトライ）の段階で設計の作り直しが必要になる可能性が高い。長時間プロセスの管理は Next.js のモデルと根本的に相性が悪く、後から分離するコストは最初から分けるコストより大きい

### CLI ツールのみ（Web UI なし）

- **メリット**: 最も軽量で開発が速い
- **却下理由**: 複数エージェントの並列実行状況やタスクの一覧性を確保するには、カンバン形式の UI が必要。CLI だけでは進捗の俯瞰が難しい

## 結果

- フロントエンドと job-manager の間の通信プロトコル（REST / WebSocket 等）を別途設計する必要がある
- 共有する型定義は `packages/shared` で管理する
- 開発時は `pnpm dev` で両サービスを並列起動する
