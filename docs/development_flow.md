# Expedition - 開発フロー設計書

## 目的

現状の開発フロー（Jira → 実装 → PR → マージ）に沿ったタスク管理を行い、
フローの各ステップでClaude Code / Devinによる自動化を実現する。

---

## 開発フロー全体像

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 1. 課題発生  │───→│ 2. タスク    │───→│ 3. 実装     │───→│ 4. PR作成・  │───→│ 5. CI確認・  │───→│ 6. マージ・  │
│   (Jira)    │    │   細分化     │    │  (AI実行)   │    │   レビュー   │    │   自動修正   │    │   完了処理   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

---

## 各ステップの詳細

### 1. 課題の発生（Jira連携）

**現状のフロー:**
- Jiraに課題が起票される
- スプリントごとに担当者をアサイン
- 課題はユーザー視点（何ができるようになるか）で記述されており、具体的な作業内容は未記載

**Expeditionが提供する機能:**
- Jira REST APIで担当課題を取得・一覧表示（カンバンUI）
- 課題の詳細表示（説明、受け入れ条件、関連課題など）

**必要な連携:**
- Jira Cloud API（APIトークン認証）
- JQL: `assignee = currentUser() AND sprint in openSprints()`

---

### 2. タスクの細分化（AI支援）

**現状のフロー:**
- 担当者がJira課題を読み、必要な作業を洗い出す
- 1タスク = 1 PR、1 PRは100行以下の変更が望ましい

**Expeditionが提供する機能:**
- Jira課題の内容をClaude Codeに渡し、作業の洗い出し・細分化を支援
- 細分化結果をサブタスクとして構造化（依存関係・実行順序付き）
- 人間が細分化結果を確認・編集してから確定できるUI

**細分化時にAIに渡すコンテキスト:**
- Jira課題の内容（タイトル、説明、受け入れ条件）
- 対象リポジトリのコードベース情報（関連ファイル、既存の実装パターン）
- チームの開発規約（PR粒度、コーディングスタイルなど）

**検討事項:**
- プロンプトの品質が細分化の精度に直結するため、テンプレート化・改善サイクルが必要
- 細分化の粒度が適切かどうかの判断基準の定義（行数だけでなく、変更の凝集度も考慮）

---

### 3. タスクの実装（AIエージェント実行）

**現状のフロー:**
- 細分化されたタスクをClaude CodeまたはDevinで実装
- Claude Code: 難易度が高い、または抽象度の高いタスク向け
- Devin: プロンプトに対して意図したコードが出やすいタスク向け
- 現時点ではエージェントの選択は人間が判断

**Expeditionが提供する機能:**
- タスクごとにgit worktreeを自動作成
- エージェント（Claude Code / Devin）への指示実行
- 複数タスクの並列実行
- 実行状態の監視・表示（実行中 / 完了 / エラー）

**エージェント選択の将来構想:**
- タスクの特性（変更範囲、テスト有無、既存パターンの有無）に基づく自動判定
- 過去の実行結果（成功率、修正回数）をフィードバックとして蓄積

**検討事項:**
- worktreeごとのDocker環境（ポート競合、DB分離）
- エージェント実行のタイムアウト設定
- 並列実行時のAPI rate limit
- タスク間のコンフリクト回避（設計段階の分割で対処するが、検知の仕組みも必要）

---

### 4. PR作成・レビュー

**現状のフロー:**
- Devin: 自動でPR作成
- Claude Code: 実装完了後、担当者がPR作成

**Expeditionが提供する機能:**
- Claude Code完了時にワンクリックでPR作成
- PRタイトル・説明の自動生成（変更内容の要約、関連Jira課題のリンク）
- PR一覧でレビュー状態を可視化（レビュー待ち / 承認済み / 変更要求）

**検討事項:**
- PRテンプレートとの整合性
- レビューコメントへの自動応答（AIによる修正提案）の要否

---

### 5. CI確認・自動修正

**現状のフロー:**
- PR作成後にCIが実行される
- Devin: CI失敗時に自動で再実装
- Claude Code: CI失敗時に手動で対応

**Expeditionが提供する機能:**
- CI結果のポーリング（GitHub Actions API）
- CI失敗時にログを取得し、Claude Codeに修正を指示
- 修正後の再push → CI再実行の自動サイクル

**安全策:**
- 最大リトライ回数の制限（デフォルト3回程度）
- 同じエラーが繰り返される場合の検知・停止
- CIログが長大な場合のトリミング（関連部分の抽出）
- 人間への通知・エスカレーション機能（Slack等）

---

### 6. マージ・完了処理

**現状のフロー:**
- PRがマージされたらタスク完了

**Expeditionが提供する機能:**
- マージ検知（GitHub webhook or ポーリング）
- Expedition内のタスクステータスを「完了」に更新
- Jira課題のステータス更新（トランジション実行）
- 関連サブタスクがすべて完了した場合、親課題のステータスも更新

---

## たたき台に対する補足・追加検討事項

### A. 通知・アラート機能
フロー全体を通じて、以下のタイミングで担当者に通知する仕組みが必要:
- AI実装が完了した
- CI失敗でリトライ上限に達した
- PRにレビューコメントがついた
- 長時間停滞しているタスクがある

通知先: Slack、ブラウザ通知、SSE（Expedition UI上）

### B. エラーハンドリング・障害復旧
AIエージェントの実行は失敗することが前提:
- エージェントがハングした場合のタイムアウト・強制終了
- worktreeの状態が壊れた場合のリセット手段
- 途中で止まったタスクの再開手段

### C. コスト管理
AIエージェントの利用にはAPIコストが発生する:
- タスクごとのトークン使用量・API呼び出し回数の記録
- 日次/月次のコストサマリー表示
- コスト上限設定とアラート

### D. 実行ログ・監査証跡
AIが生成したコードの品質管理・振り返りのために:
- エージェントへの指示内容と応答の記録
- 各タスクの実行時間、リトライ回数、最終結果の記録
- 成功/失敗パターンの分析（プロンプト改善のインプット）

### E. 権限・セキュリティ
- AIエージェントに渡す情報のスコープ制御（機密情報の除外）
- APIトークンの安全な管理（環境変数 or シークレットマネージャー）

### F. Jira課題 → サブタスク間のトレーサビリティ
- Jira課題とExpedition内サブタスクの紐付け管理
- 1つのJira課題に対する進捗率の可視化（完了サブタスク数 / 全サブタスク数）
- Jira課題のステータス更新タイミングの定義（全サブタスク完了時 or 段階的に）

## タスクのステータス遷移

```
[未着手] → [細分化中] → [実装待ち] → [実装中] → [PR作成済] → [CI実行中] → [レビュー待ち] → [マージ済] → [完了]
                                          ↑                        │
                                          └── [CI失敗・修正中] ←───┘
```

- 各ステータスは自動遷移を基本とし、手動遷移もUIから可能にする
- ステータス遷移はExpedition内で管理し、Jiraへは主要なステータス（着手/レビュー中/完了）のみ同期
